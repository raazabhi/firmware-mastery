# The Definitive Guide: Inline Debugging (Beginner to Pro)

In embedded software, writing the code is only 20% of the job. The other 80% is figuring out why the hardware isn't doing what your code told it to do. This guide tracks the evolution of an embedded engineer's debugging arsenal.

## Level 1: The Beginner (The `printf` Developer)
**The Toolkit:** UART strings and LED blinking.

* **How it works:** You litter your code with `printf("Got here 1\n");` and watch the terminal.
* **The Pitfall:** `printf` over UART is incredibly slow (often taking milliseconds). By the time the string prints, the interrupt timing has shifted, the I2C bus has timed out, and the bug disappears (a "Heisenbug").
* **When to use:** High-level state machine logging (e.g., `[INFO] WiFi Connected`).

## Level 2: The Intermediate (The GDB Stepper)
**The Toolkit:** Hardware Debugger (ST-Link, J-Link), OpenOCD/GDB, Breakpoints.

* **How it works:** You set a red dot on a line of C code. The CPU literally stops executing when it hits that line. You inspect variables and use "Step Over" (F6).
* **The Pitfall:** The CPU stops, but the physical world *doesn't*. If you halt the CPU while a motor is spinning, the PWM might stay stuck on HIGH, frying the motor. If you halt during an I2C transaction, the slave will time out.
* **When to use:** Logic verification, checking buffer contents, reading static hardware registers.

## Level 3: The Pro (The Non-Intrusive Tracer)
**The Toolkit:** SWO (Serial Wire Output), Data Watchpoints (DWT), Logic Analyzers.

* **SWO Tracing:** Instead of slow UART `printf`, you use the ITM (Instrumentation Trace Macrocell) built into the ARM Cortex core. It sends `printf` strings directly over the JTAG/SWD cable at speeds of up to 2 Mbits/s with *zero* CPU overhead.
* **Data Watchpoints:** Instead of breaking on a *line of code*, you break on a *memory address*. If a random pointer is corrupting your `SystemState` variable, you set a Watchpoint on `SystemState` for "Write". The CPU will halt the exact nanosecond the rogue pointer strikes, revealing the culprit in the Call Stack.
* **Logic Analyzers:** You don't trust the CPU. You hook $15 logic analyzer probes to the physical TX/RX or SDA/SCL pins to see the actual electrical square waves.

## Level 4: The Hardcore Pro (The Post-Mortem Analyst)
**The Toolkit:** ETM Instruction Tracing, HardFault Assembly Wrappers, Map Files.

* **HardFault Handlers:** The CPU crashes. The Pro doesn't just reboot. They look at the `LR` (Link Register) on the stack, open the `.map` file generated by the compiler, and map the raw hex memory address back to the exact line of C code that caused the crash.
* **ETM (Embedded Trace Macrocell):** Requires a special 20-pin trace debugger. It records every single branch (if/else) the CPU takes in real-time without slowing it down. When the system crashes, you can literally "rewind" the code execution line-by-line to see exactly how it arrived at the crash.
